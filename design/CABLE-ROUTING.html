sigpath

Cable routing — one general orthogonal router. Replace the five horizontal-only passes with a single router that routes every cable — including the ethernet/USB runs the current pipeline drops on the floor — around devices, with guaranteed clearance and a tunable bias toward legibility. The design behind roadmap items p2-bidiroute, p2-router, p2-makeroom, and p2-trunk.

Last updated June 25, 2026 · ← Product roadmap · Zone ↔ tab → · Signal-grade → · github.com/patricktr/sigpath

The one idea

  • One router behind one frozen seam. Everything between displayEdges and the screen becomes a pure function route(req) → Map<edgeId, Pt[]>. Old pipeline and new router both implement it, so they are swappable behind a flag and the rest of the render path (validation styling, gradients, the cable-ID badges, the jogOffset nudge) never moves.
  • Boxes are a hard wall; crossing a cable is a soft cost. A device's interior and a clearance gutter around it are never entered — that's an inviolable A* constraint, not a penalty. Crossing another cable is just a weighted cost, traded against bends and length. This is exactly your “never behind a device, never adjacent; minimize crossings when you can” split, made literal.
  • One joint cost, one legibility slider. Today crossings and bends are minimized in separate passes with no common currency, so “bias toward legibility” can't even be expressed. The new router scores every route by w·length + bends + crossings at a fixed ~2:1 crossing:bend ratio, surfaced as a single slider.
  • Don't trust the rewrite — measure it. A prior attempt to generalize the router regressed crossings 1→3 and was reverted. So non-regression is a gate, not a hope: one canonical crossing counter, a tracked fixture corpus, and a parity-or-better bar that the default flip cannot pass without. The new router earns each case before it owns it.
  • Placement bounds routing. Real pro-AV sheets get their clarity from aligned device columns with clean vertical channels between them — not from a clever path through a cramped layout. So “Make room” (nudge devices to open channels) and trunk bundling are first-class follow-ons, not afterthoughts.

1Today: five passes, and a gate that drops half the cables

All cable geometry is computed in one React useMemo — displayEdges (App.tsx:518–703) — that runs five sequential passes per render and hands each edge its interior bend points via data.waypoints. CableEdge.tsx draws that rounded orthogonal path, or — if a run carries no waypoints — falls back to React Flow's getSmoothStepPath.

  1. Collect obstacles. Every port-bearing node (device, block) becomes a rect at measured-or-estimated size; zones/notes join only if data.obstacle is set (App.tsx:530–544).
  2. Resolve endpoints — and gate. For each edge, resolve its two ports, then App.tsx:558: if (sp?.direction !== "output" || tp?.direction !== "input") continue; Only a right-side output → left-side input run survives. Everything else is dropped here.
  3. Detour blocked runs. routeAroundObstacles (obstacleRoute.ts:108): a fewest-bends A* over a sparse Hanan grid — hardwired horizontal (exit +x, enter −x, obstacles filtered to the x-corridor, TURN_COST=100000 to minimize bends).
  4. Fan parallel runs into lanes. spreadDetourBundles + assignLanes (parallelLanes.ts:184) cluster nearby runs and pick a crossing-minimizing track order — but only within a cluster.
  5. Nudge collinear overlaps. nudgeCollinearOverlaps (obstacleRoute.ts:646) fans any two different cables sharing a line apart by NUDGE_GAP=6 with a small port hop.

For the common case — a right-output to left-input run — this is genuinely good, and finely tuned. The problem is everything it doesn't cover, and a structural ceiling on what it can express:

2The calls you made

Research surfaced four directions and several cost/benefit forks; these are the answers that drive the rest of the design. Two of them follow directly from your own words and were locked without asking.

Build a general router

Replace the pass-by-pass pipeline with one self-built general orthogonal router — not a patch, not a library (no libavoid/WASM, no LGPL; stays MIT and local-first).

your pick · Q1 + Q2

Opt-in “Make room”

Auto-nudging devices to open channels ships as an opt-in “Make room / Tidy” command — grid-snapped, previewed, undoable, never silent.

your pick · Q3

Offer trunks at ≥4

≤3 like-cables stay fanned; at ≥4 sharing a corridor, offer (never force) a collapsible labeled trunk with a count badge. Per-cable detail survives.

your pick · Q4

Hard wall, soft crossing

Box interior + clearance gutter = hard constraint; crossing a cable = soft penalized cost. Derived from “never behind/adjacent” (absolute) vs “minimize crossings when they can” (soft).

locked from your brief

Legibility = a 2:1 slider

A joint cost w·len + bends + crossings with a tunable ~2:1 crossing:bend ratio, surfaced as one slider. Materializes only because there's now a unified router to host it.

locked from your brief

The hard constraint

The tuned horizontal output→input case must not regress. So the rewrite is phased and gated — see §4. This is the one rule everything else bends around.

non-negotiable
The scope resolution. “Full rewrite” and “build small/scoped” aren't in tension once the rewrite is phased: we build one general router, but it first owns only the cases that are unrouted today (where there is nothing to regress), then expands to own everything once the parity gate passes, then becomes the default, then the legacy code is deleted. Ambition and safety, same plan.

3The router: one two-phase orthogonal engine

A new module src/flow/router/ exporting routeAll(req): RouteResult. It is a textbook two-phase orthogonal connector router (visibility/Hanan graph + A*, then a nudging pass) specialized to this app, replacing legacy stages 2–5. It is a pure function of geometry — no React Flow or DOM types cross its boundary.

3.1 — Geometry in, no DOM out

App builds a RouteCtx once per render and passes it in. Every box is inflated by the clearance gutter; no graph vertex or segment may lie in any inflated interior (the existing strict segmentHitsRect, obstacleRoute.ts:62, reused verbatim). 

type PortGeom = {
  nodeId: string; portId: string;
  side: 'L' | 'R' | 'B';     // left input · right output · bottom bidi
  anchor: Pt;                  // the exact jack position on the canvas
  stubEnd: Pt;                 // anchor + PORT_STUB along outDir (perpendicular exit)
  outDir: '+x' | '-x' | '+y';  // L:-x  R:+x  B:+y
};
The bottom-port trap (critique fix). Left/right ports sit on the 24px grid rows, so their anchor is computed from the rect. Bottom (bidi) ports are different. The CSS is .device-node__io { display:flex; flex-wrap:wrap; gap:8px 14px; justify-content:center } — each port a label-width-driven column, ports wrapping to multiple rows. No fixed-pitch “centered, 14px apart” formula can place them; a formula would attach cables beside the jack. So App reads each bottom handle's measured center from React Flow's handleBounds and passes it in as anchor. The router never synthesizes bidi geometry. And any run touching a bidi port is routed only once the node and its handles are measured — until then it keeps the gate's old behavior for that frame, because the port-count size estimate omits the bottom IO bank and would under-cover the box.

3.2 — Phase 1: a shared grid + direction-aware A*

Build one sparse Hanan grid over the full bounding region (not a per-run x-corridor — that corridor filter is exactly what starved vertical travel of gridlines and let the last attempt cut through stacked boxes). X gridlines: inflated box edges, port-stub columns, and a channel midline between each horizontally-adjacent box pair (this subsumes centerDetourVerticals). Y gridlines mirror it. Total gridlines are capped, with a box-edge-only fallback above the cap to bound the O(B²) midline blow-up.

  1. No privileged axis. A* state is (xi, yi, dir∈{H,V}), reusing the existing binary heap (obstacleRoute.ts:150–179). A mandatory first move anchor→stubEnd along outDir gives every port a clean perpendicular exit on all four sides; the goal is accepted only on an axis-matched final move into the target anchor.
  2. One joint cost. Per expansion: w_len·len + w_bend·(turned?1:0) + w_cross·estCross(seg). Box-piercing segments are simply never expanded (hard wall). The heuristic uses only the length term, so it stays admissible and the optimal-bend guarantee today's TURN_COST relies on is preserved.
  3. Two sweeps, crossing-aware. 1a routes every run with w_cross=0 (pure bend+length) to get provisional polylines and build per-gridline occupancy. 1b — mandatory for any run sharing a congested corridor — re-routes with the crossing penalty active, seeded from the previous frame's committed routes so route choice (not just track order) avoids crossings and stays temporally stable.

3.3 — Phase 2: channel order + 1-D separation

One unified pass replaces all four of today's geometry passes (assignLanes / spreadDetourBundles / nudgeCollinearOverlaps / centerDetourVerticals). Extract every axis-aligned segment; union-find cluster the co-linear, span-overlapping ones; per cluster, order to minimize real crossings (the proven brute(≤7)/greedy(≤20) ladder from bestOrder, parallelLanes.ts:129), then separate into evenly-spaced, non-adjacent tracks using spreadDetourBundles' flanking-wall room logic. A jogOffset-pinned run is an immovable track the others order around. This is a per-cluster greedy 1-D projection — no VPSC solver, no new dependency.

3.4 — The one render edit, and one canonical crossing count

CableEdge.tsx:52–53 hardcodes pts[1].y = sourceY / pts[len−2].y = targetY — the horizontal-exit snap. Make it axis-aware off the sourcePosition/targetPosition already in EdgeProps: snap Y for left/right exits, snap X for top/bottom. The label placer already handles bottom (CableEdge.tsx:82–86), so bidi cable IDs work the moment the path does.

One crossing function, measured on the same geometry. Three crossing definitions exist today. The router's internal objective and the parity gate must call the same canonical counter on the same post-stitch polylines — otherwise v2 can pass/fail the gate for the wrong reason. Since the port-snap can move a near-port segment and create or remove a crossing, the router runs the stitch as a pure function before counting.

5Make room / Tidy p2-makeroom

Not auto-layout. It preserves your arrangement and only widens vertical channels where the router reports cables can't keep clearance — by small, grid-snapped column shifts — then reroutes. The biggest single legibility lever, because a clean route needs a clean channel to live in.

  1. Detect. Cluster device left-edges into columns; an adjacent column pair is a channel. It's congested when demand > supply, or clearance violations touch it, or a lane gap was compressed. The router emits a small RoutingReport for this; a polyline-derived fallback (count vertical segments per gap) means Make-room is never blocked on it.
  2. Propose. Shift the right-hand column group right by min(neededWidth, MAX_TIDY_SHIFT); report “partially tidied” when capped. The source column never moves. Zones ride along as rigid units via nodesInZone (the same primitive drag uses), assigned whole to their majority side.
  3. Legalize. Snap moved X to the 24px grid; one overlap sweep enforcing a 2×gutter minimum gap.
  4. Crossing guard. Recompute total crossings on the proposal; refuse any proposal that increases them. The locked non-regression rule applies to Make-room too.
  5. Preview & commit. Ghost positions + highlighted channels + “Make room for N cables in M channels — Apply / Cancel.” Apply goes through takeSnapshot() (one undo step). No congestion → no-op + status, no snapshot.

6Trunk bundling p2-trunk

At ≥4 like-cables sharing a source-region → dest-region corridor, offer one orthogonal labeled spine with a count badge and expand/collapse. A derived candidate per render; a persisted entity only once accepted.

Detect & offer

Group edges by (signalKind, sourceRegion, destRegion); cluster by shared corridor (reusing the assignLanes same-corridor test). A cluster of ≥4 raises a non-modal chip: “Bundle 4 audio cables into a trunk?” Dismissals are remembered so it never nags.

Route as one spine

A collapsed trunk is one virtual edge through the general router — members suppressed from router input — so the backbone obeys the same box+gutter wall and counts in the global crossing total. Numbered fan-out stubs run from each real port to the spine, crossing-minimized.

Traceability survives

Each member keeps its own Connection untouched; the BOM and pack list (deriveLists) read connections directly and are unaffected. Selecting the badge highlights members; selecting a stub selects its cable.

Persistence schema v6

Additive Diagram.trunks? keyed by connection id; round-trips through serialize both directions and the revision snapshot/hash — fixture-gated so a saved trunk survives save/load and restore. Old files load unchanged.

7The plan: seven phases, gated

Each phase has a crisp exit criterion measured on the corpus. The router only widens its scope after the gate for its current scope passes. Roadmap ids in the right column.

PhaseGoal & exit criterionEffortRoadmap
P0 · Lossless lift Extract the legacy pipeline behind interface Router + pickRouter(), zero behavior change. Exit: byte-identical waypoints on all fixtures. M —
P1 · Metric + corpus One canonical crossing counter, routeMetrics.ts, the headless harness, the 6-fixture corpus, shadow-mode overlay. Exit: legacy baselines recorded; counter unit tests pass. M —
P2 · Bidi/bottom The new router owns only the dropped runs; delegates output→input to legacy. Fixes the box-cut bug. Exit: on bidi-bottom, zero box-interior hits AND crossings ≤ legacy. XL p2-bidiroute
P3 · Own all cases Extend to output→input with the joint cost, unified Phase 2, jog pins. Exit: parity-or-better on all 6 fixtures + visual diff + a profiled per-render budget at 100 devices / 200 cables. XL p2-router
P4 · Flip default Make v2 the default (legacy reachable ≥1 release); emit the RoutingReport. Exit: no open regression after a soak; report validated against the metric. M p2-router
P5 · Make room Opt-in channel widening with preview + undo + crossing guard. Exit: widens congestion without raising crossings; zone membership invariant; single undo. L p2-makeroom
P6 · Trunks Offer + collapse + persist (schema v6). Exit: a saved trunk survives save/load and revision restore; BOM unchanged; spine obeys box+gutter and counts in crossings. L p2-trunk
The first shippable win is P2. P0–P1 are scaffolding with no user-visible change; P2 fixes the ethernet/USB-through-boxes bug for real, and by construction it can't touch the horizontal case (those runs are still delegated to legacy). Everything riskier waits behind the parity gate.
