sigpath

Cable schedule + BOM export. Turn the connections already on the canvas into the deliverables a crew actually carries to site — an enriched port-to-port cable schedule and an equipment bill of materials — exportable as a printable PDF, an Excel workbook, CSV, and physical cable labels, plus an in-app install checklist that tracks each run from planned to tested. The design behind roadmap item p3-cableschedule.

Last updated July 1, 2026 · ← Product roadmap · Cable routing → · Zone ↔ tab → · github.com/patricktr/sigpath

The one idea

  • The data already exists — this feature is formatting, not computing. deriveLists(nodes, edges) already yields a patches port-to-port table and a device pack list. Every export is a pure view over one DerivedLists; nothing re-walks the graph. Enrich that struct once, format it five ways.
  • Enrich once, format many. Widen PatchRow with the two end connectors, colors, and a per-cable note a single time. The Lists panel, PDF, XLSX, CSV, clipboard, and label sheet all read the same enriched rows — no per-format graph logic, no drift between what you see and what you export.
  • Meters in, chosen unit out. Lengths stay stored in meters (see units.ts / p2-prefs); every schedule surface renders through formatLength(m, unit). One rounding point, at the edge, so PDF/XLSX/CSV all agree with the on-screen table and with each other.
  • Install state travels with the project. Checklist status is job state, not a UI preference — it persists in the .sig file (additive schema bump v6 → v7) so a shared diagram carries its progress. It marks the project dirty but stays off the undo stack — checking off a pulled cable is not a design edit.
  • Technical, not commercial. The BOM is quantity + manufacturer/model. No pricing, vendors, or part numbers — sigpath stays a design tool, not a quoting app. Spare/overage math is a separate, already-scoped item (p3-bomrules).

1Today: the data is here, the deliverables aren't

3Data model & persistence

Three additive fields. Two ride on the cable edge; one is diagram-level. All round-trip through serialize.ts behind a single schema bump.

// src/flow/types.ts — CableEdgeData gains two persisted fields
type CableEdgeData = {
  cableTypeId: CableTypeId;
  number?: string;          // existing cable ID (AV-001)
  lengthMeters?: number;    // existing
  note?: string;            // NEW — free-text schedule note
  install?: InstallStatus;   // NEW — "planned" | "pulled" | "terminated" | "tested"
  /* …grade, jogOffset, derived render fields unchanged… */
};

// Diagram-level: received/installed counts, keyed by device model id.
// Lives beside `boundary` and `trunks` on EditorDiagram, like other job metadata.
type EditorDiagram = { /* … */ bomProgress?: Record<string, number> };
  • Where install status renders in the schedule. A cable with no install is treated as planned (the default), so old diagrams read correctly with zero migration.
  • Undo. note edits go through the normal snapshot path (a design edit). install and bomProgress mutations bypass the undo stack — they set dirty and persist, but Ctrl-Z during a design session never rewinds site progress.
  • Schema. SIGPATH_SCHEMA_VERSION 6 → 7, additive and backward-tolerant, exactly as p2-trunk did at v6. Serialize writes the new fields when present; deserialize defaults them when absent.

Print, not just generate. The label sheet and (optionally) the PDF use a dedicated print stylesheet + window.print(), letting the OS handle "Save as PDF" and real paper — cheaper and higher-fidelity than pixel-painting a canvas.

6Interactive install checklist

A mode on the Lists panel that turns the read-only schedule into a live site tracker. Two independent trackers, one per list:

  1. Cables — a status pipeline. Each run cycles Planned → Pulled → Terminated → Tested via a click-through pill (or a select). The status writes to edge.data.install. A per-signal-group progress readout ("SDI 6 / 8 terminated") sits atop the table.
  2. Equipment — received vs needed. Each BOM line shows needed (the derived count) and an editable received (bomProgress[modelId]). A line is "complete" when received ≥ needed — the user's "QTY out = QTY needed".
  3. Hide-completed filter. One toggle collapses done runs / satisfied BOM lines so what's left on site is what's left to do. Ephemeral view state, like the signal filter (p2-typefilter).
Scope guard. Status is a flat per-cable field, not a workflow engine — no assignees, timestamps, or test-result attachments (those belong to a commissioning tool). If install mode starts growing verbs, that's the signal it's a separate feature.

8Deferred & adjacent

  • Spares / overage — "pack one extra per N", round lengths to stock: its own roadmap item, p3-bomrules. The BOM here stays raw counts; that layer adds traceable spare lines on top.
  • Custom connector-pair cable types (p4-cabletypes) — once a cable carries its own two end connectors, the A/B-end columns read from the cable type instead of the mated ports. The schedule columns are chosen to absorb that with no shape change.
  • Physical location / rack columns — a real "from rack → to rack" needs a rack/room model sigpath doesn't have yet. Explicitly out of scope; the device label is the location proxy until then.
  • A4 page size, label-stock presets, price fields — Preferences toggles / later polish, not blockers for A–D.
