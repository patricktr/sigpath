Zone ↔ tab — nested sub-diagrams. A zone and a tab are two views of the same thing: edit a control room in detail as its own tab, then drop it into the event diagram as one wired block that exposes its boundary ports. The design behind roadmap item p2-zonetab.

Last updated June 24, 2026 · ← Product roadmap · Signal-grade architecture → · Community architecture → · github.com/patricktr/sigpath

The one idea

  • A tab is the content; a block is a view of it. A referenced Diagram is the single source of truth. Each placement is a BlockInstance — a first-class node that renders that diagram's published boundary ports as connectable handles. Build a room once, wire it in many places.
  • A block is not a zone. A Zone stays a decorative rectangle. The block is a disjoint fourth node kind that reads unmistakably as “a reference to a tab,” never an overloaded rectangle that sometimes means something else.
  • The schema barely moves; the consumers do. v3→v4 is strictly additive (old files load untouched). The real work is that validate, the pack list, cable numbering, and export all assume one flat diagram today — so nesting threads through a single shared nodePorts() seam instead of a scattered rewrite. That is the cost, and it only grows the longer we wait.
  • Boundary ports are Port-shaped and stable by construction. They carry a connector and grade like any port, so connector-compat and the grade gate run across a boundary with no special path — and their ids are minted once, so internal refactors never silently rewire a block.

1Today: a flat list of diagrams

A project is SigpathDocument → Project { diagrams: Diagram[] } → Diagram { devices, connections, zones, annotations }. The tree is exactly three levels deep and there is no way to nest one diagram inside another. Three facts make nesting a real piece of work rather than a flag:

A Zone is decorative

Zone (document.ts:15) is a zIndex:-1 rectangle with a label, color, and an obstacle flag. It links to nothing.

Ports live only on devices

A PortRef (connection.ts:7) is {instanceId, portId} where instanceId is a DeviceInstance.id in the same diagram. There is no notion of a diagram-level port.

Every consumer gates on device

validate.ts:91, derive.ts:41, cableId.ts:25 all branch on n.type === "device" and resolve endpoints via node.data.model.ports.find(...).

On the editor side, useProject owns diagrams[] + activeId, and only the active diagram is live — its nodes/edges sit in React Flow state and fold back via synced() (useProject.ts:46); every other diagram is a frozen snapshot. The undo unit is the whole project. serialize.ts maps device↔DeviceInstance and edge↔Connection and silently drops any other node type. And parseDocument reads schemaVersion but never branches on it — every change v1→v3 was additive-optional, so there is no real loader yet. Nesting is the first feature that needs one.

2Four ways to model nesting

Four independent designs were weighed by three judge panels (architecture, migration, product). The winner is B, with the best ideas from the other three grafted in — not a single-approach purity play, but the synthesis the panels converged on.

B · Dedicated block node adopt

A first-class BlockInstance references a Diagram and renders its boundary ports. Zone stays cosmetic; the block reads unmistakably as a reference. Stable mint-once port ids; one shared seam fixes all consumers; routing kernels need zero change because a block adopts the device left-in / right-out handle convention.

adopt — the spine

C · Published interface graft

A diagram declares an explicit, version-stamped port contract. Highest conceptual integrity and the only model where a refactor provably can't silently change a block's ports — but it front-loads a publish step and the codebase's first non-trivial loader. Grafted: its rev-stamp drift state and its “cables to a removed port show Broken, never silently dropped” rule.

graft the rev-stamp

D · Block-as-device proxy graft

Lowest blast radius: a block satisfies data.model.ports verbatim. But making a block pretend to be a device leaks garbage into the BOM and forks counting. Grafted: the synthesized DeviceModel as an internal resolve/render shim behind nodePorts() — never a user-facing masquerade.

graft the shim

A · Zone-as-reference rejected

Smallest schema delta — Zone gains refDiagramId + a cached boundary list. But it overloads one type to mean both rectangle and live reference (mode confusion), and the cache is denormalized state that rots on every sub-diagram edit. The “minimal” framing is hollow — it still needs the full flatten pass. Grafted: auto-expose on promote.

rejected as primary

3The model: a block is a reference to a tab

The referenced Diagram is the single source of truth for both its content and its published boundary set. A BoundaryPort is deliberately Port-shaped (id / name / direction / connector / accepts? / grade?) plus an internal: PortRef back-pointer to the real inner device port it proxies. Each placement is a distinct BlockInstance with its own id and position, sharing refDiagramId.

Control Room · its own tab
Switcher → Multiviewer → … real devices, real internal cables
↓  publish the crossing ports
boundary.ports: PGM Out, Tally In Port-shaped · mint-once id · internal → inner device port
Event · the host tab
[ Control Room ] block renders PGM Out (right) · Tally In (left) as Handles
↓  an ordinary host-diagram cable
[ Backstage ] block itself another tab, embedded the same way

The trick that keeps the blast radius small: the block node carries a synthesized DeviceModel whose model.ports are the referenced diagram's boundary ports. So node.data.model.ports.find(...) resolves for a block exactly as for a device — the four consumers widen one type guard behind a shared nodePorts() seam instead of forking their logic. PortRef and Connection are byte-unchanged on disk: a top-level cable lives in the host diagram's connections[], its endpoint instanceId is a BlockInstance.id and its portId a BoundaryPort.id (which is also the Handle id the block renders).

4Schema (v4, additive)

Bump SIGPATH_SCHEMA_VERSION 3 → 4. Every new field is optional, so v1–v3 files load unchanged with no transform. Zone and PortRef are not touched. 

// src/schema/boundary.ts (new)
import type { PortDirection } from "./device";
import type { ConnectorId } from "./connectors";
import type { GradeId } from "./grades";

/** The outward face a Diagram exposes when embedded as a block. Port-shaped so
 *  compat/grade/cable-prefix work across a boundary with no special path. */
export type BoundaryPort = {
  id: string;            // STABLE, minted once; survives internal rename/renumber/rebind
  name: string;          // editable display label, e.g. "PGM Out"
  direction: PortDirection;
  connector: ConnectorId;     // mirrored from the inner port at expose-time
  accepts?: ConnectorId[];
  grade?: GradeId;            // mirrored from the inner port
  /** The single internal device port this boundary proxies. */
  internal: { instanceId: string; portId: string };
};
// src/schema/document.ts — Diagram gains an optional published interface + placed refs
export type Diagram = {
  id; name; devices; connections; zones; annotations; orientation?;
  boundary?: { ports: BoundaryPort[]; rev: number }; // absent ⇒ not embeddable
  blocks?: BlockInstance[];                           // placed references to other diagrams
};

/** A placed reference to another diagram, rendered as a block. NOT a Zone. */
export type BlockInstance = {
  id: string;                 // unique within the HOST diagram
  refDiagramId: string;
  label?: string;             // per-placement override; default = ref diagram name
  position: { x: number; y: number };
  size?: { width: number; height: number };
  obstacle?: boolean;         // routing opt-in; default ON for blocks
  boundaryRev: number;        // the ref diagram's boundary.rev this block was bound to
};

export const SIGPATH_SCHEMA_VERSION = 4;
// src/flow/types.ts — new node type + widened union
export type BlockNodeData = {
  refDiagramId: string;
  label?: string;
  model: DeviceModel;   // SYNTHESIZED from the ref diagram's boundary.ports at load/render
  boundaryRev: number;
  drift?: boolean;      // derived: boundaryRev < ref diagram's live boundary.rev
  obstacle?: boolean;
};
export type BlockNodeType = Node<BlockNodeData, "block">;
export type SigNode = DeviceNodeType | ZoneNodeType | NoteNodeType | BlockNodeType;

5Boundary ports — stable by construction

The BoundaryPort is the single genuinely new concept, and it earns its keep three ways:

  1. Port-shaped, so no parallel compat path. Because it carries connector and grade, checkPortCompatibility (connection.ts:148), the grade gate (validate.ts runDemand), and cablePrefix (cableId.ts:36) all run on a boundary port through the shared nodePorts() seam with zero special-casing.
  2. Source of truth on the tab, not cached per block. Boundary ports live on Diagram.boundary.ports. Each block synthesizes its model from the live referenced diagram at load/render (read through synced() when that tab is active so an in-flight edit isn't stale). No denormalized copy to rot.
  3. Mint-once ids + a semantic sticky key. The id is minted once and survives internal rename/renumber/rebind. Re-derivation after an inner edit matches on the semantic tuple (internal.instanceId + direction + connector + ordinal) — never the unstable inner port id — so the inspector replacing a device's model wholesale (App.tsx:875) can't orphan every top-level cable.

6Cross-diagram connections — no new edge type

There is no diagram qualifier on PortRef and no new edge kind. The cross-diagram link is always the explicit block → boundary → internal hop, and endpoint resolution is always scoped to the connection's own diagram.

A cable that touches a block is structurally an ordinary host-diagram edge whose source or target happens to be a block node — so undo/snapshot, jogOffset, lane assignment, and manual routing overrides all keep working with no kernel change. The only place two diagrams' ids ever coexist is inside flatten(project, rootId), which namespaces every inner id per block-instance path before any project-wide pass (oversubscription, duplicate-id, BOM counts, cable renumber). Because device ids are preserved when promote moves them into a new tab, the same id can legitimately exist in two diagrams — which is safe precisely because resolution never crosses a diagram boundary except through a boundary port.

One physical cable stays one cable — a boundary port is a projection, not a patch point. A boundary port is the outward face of exactly one inner device port (its internal ref); you never run a separate “device → boundary” wire inside the sub-tab. So a run like Switcher PGM → [Control Room] → Projector is a single Connection in the host diagram ((block, PGM Out) → (Projector, in)); flatten() rewrites the block endpoint to internal, yielding one edge (Switcher, PGM) → (Projector, in) — so the BOM and cable numbering see one cable, not two. A real physical patch panel / wall plate / tie-line (where a cable genuinely terminates and another begins) is modeled as an actual device, not a boundary — see decision 7.

7Promote ⇄ embed

The two verbs that make a zone and a tab interchangeable. Each is one atomic takeSnapshot in useProject.ts.

promoteZoneToTab(zoneId) — zone becomes its own tab

  1. Sync the host first. Derive both diagrams from one synced() value so the move is a single atomic commit (no split-snapshot orphans).
  2. Compute membership, preview & confirm. Geometric containment of the zone rect, shown for the user to add/remove before committing — promote is destructive.
  3. Move contained devices/notes/internal cables into a fresh tab. Ids preserved, positions rebased. They now live only in the sub-diagram.
  4. Auto-publish a boundary port per crossing cable. Mint id, mirror connector/grade/direction, internal = the inside endpoint, rev = 1.
  5. Replace the zone with one BlockInstance, re-point crossing cables. Each crossing cable's inside endpoint becomes {blockId, boundaryPortId}. Toast + undo make the destructive move recoverable.

embedTabAsBlock(refDiagramId, position) — drop a tab in as a block

  1. Cycle guard. DFS over blocks[].refDiagramId rejects self-embed and any back-edge (A→B→A).
  2. Auto-expose if no boundary set yet. Publish dangling device ports; prompt if none dangle.
  3. Insert one BlockInstance — nothing moves. The same tab can be embedded N times as distinct instances sharing one boundary id set. Paste/duplicate routes through this same function so the guards always run.

8Rendering the block

A new BlockNode component, registered in nodeTypes (App.tsx:106): a titled box with an “opens-tab” glyph and the house-style dashed/accent border that already signals a new node — visually unmistakable from a device.

Boundary ports render as Handles (inputs Left, outputs Right, bidirectional as paired source+target on the bottom), with Handle id === BoundaryPort.id. Double-click switches to the referenced tab. Geometry gates on measured + updateNodeInternals so a freshly inserted block doesn't mis-route at the 168px default before React Flow re-measures. Because the synthesized DeviceModel satisfies the existing contract, displayEdges / buildCableEdge / lanes / jogOffset / obstacle-routing treat a block-touching cable as an ordinary edge — zero kernel change.

10Migration — and the first real loader

Strictly additive v3→v4: bump the version, extend the doc comment, ship. All new fields optional; v1–v3 files load unchanged; PortRef is byte-identical.

Nesting is, however, the first feature with cross-references that can dangle — so it earns sigpath's first non-trivial loader: a normalizeDocument(doc) pass run for all versions between parseDocument and fromDocument. It rebinds or flags a missing refDiagramId to a cable-preserving placeholder, leaves unbound boundary ports rather than deleting them, breaks embed cycles, asserts every Connection endpoint resolves locally (fail loud, not silent), tolerates version > 4, and — while we're here — fixes the pre-existing empty-diagrams[0] crash at useProject.ts:175. Save always writes v4; there is no down-migration.

12Decisions for Patrick

Seven forks — ruled on June 24, 2026; the spec below is locked. Each lists the options and the call.

Promote membership

How is “inside the zone” decided? (a) pure geometric, one-shot; (b) geometric + interactive preview; (c) build real stored zone membership first (the p2-movewithzone groundwork).

✓ Decided — (b) preview-and-confirm. Promote is destructive (devices leave the host); a half-overlap misassignment silently loses trust. Don't block nesting on (c).

Contained devices: move or copy?

On promote — (a) MOVE: devices live only in the sub-diagram, block is a pure reference; (b) COPY: devices stay in the host too.

✓ Decided — (a) MOVE. Copy desyncs two truths and double-counts the BOM. The tab is the content; the block is the view. Communicate via toast + undo.

Boundary-port exposure

(a) auto every dangling port; (b) explicit pin per port; (c) auto-on-promote + explicit curation after.

✓ Decided — (c) auto-on-promote + curation. Auto at the promote/embed moment (the crossing cables are the obvious hookups), then a rename/hide/reorder panel keeps the room's public face intentional.

Per-instance customization

N embeds of one tab — (a) tab-level boundary only (all instances share one id set); (b) per-instance overrides (room A runs 12G, room B 3G on the same port).

✓ Decided — (a) tab-level for v1. Shared ids make N-embed identity + flatten namespacing tractable; per-instance overrides are real but separable, and unsupported in v1.

Deleting a referenced tab

(a) hard-refuse; (b) refuse with a “cascade-delete N blocks too?” confirm; (c) allow + degrade blocks to placeholders.

✓ Decided — (b) cascade-confirm gate + (c) placeholder degradation via normalizeDocument for hand-edited files. Today deleteDiagram filters with zero ref check — close that in Phase A.

Cross-boundary validation depth

(a) single-hop: validate the host cable against the boundary port's mirrored connector/grade; (b) deep: propagate demand through the boundary into the inner run.

✓ Decided — (a) single-hop for v1. Mirrored connector/grade make compat + the grade gate meaningful at the boundary without recursion; deep propagation (b) is now its own roadmap item, p2-deepgrade.

Boundary: pass-through or patch panel?

Does a room edge merge a run into one cable, or split it? (a) pass-through: a boundary port projects one inner device port — one physical cable crosses it; (b) patch point: the boundary is real hardware where one cable ends and another begins (two cables).

✓ Decided — (a) pass-through for v1: one run = one cable in the BOM (see §6). A genuine patch panel / wall plate is modeled as an actual device (it has a SKU and belongs in the BOM), keeping the boundary abstraction logical.

13Open questions & risks

Open questions

  • Phase-A lovability. With promote+embed in B, Phase A ships no user-visible verb. Should the first shippable slice be re-cut around embedTabAsBlock only (non-destructive, no flatten), with promote and N-embed flatten following — so slice one is demoable?
  • Bidirectional boundary ports. A network crossing is commonly bidirectional; the paired source+target handle sharing one id needs a validate oversubscription rule (line 234 treats only direction==='input' as always-error). Confirm the intended semantics for a bidi boundary wired twice at the host.
  • flatten() performance. validate + derive run on every change (App.tsx:482–484). For deeply nested / many-embed projects, is per-change full flatten acceptable, or do we memoize the flattened graph keyed on a content hash?
  • Cable-number determinism. renumberCables numbers in array order (cableId.ts:59). Across save/load the same project must renumber identically for a printed patch sheet — do we need a stable sort key in flatten before renumbering?
  • Export gap. In v1 the CSV BOM lists inner devices invisible on the image/PDF (which shows a block-as-box). Acceptable for v1, or does the image need a per-block “see room” caption with a device count?
  • Empty interface. Should a finished sub-diagram with no dangling internal ports be embeddable at all (it yields an empty boundary), or must such a tab explicitly publish ports first?
