sigpath

Community library — architecture. How the shared equipment database is distributed, contributed to, corrected, and moderated — while keeping hosting near $0 and contributing account-free. The design behind roadmap Phase 4.

Last updated June 19, 2026 · ← Product roadmap · github.com/patricktr/sigpath

The one idea

  • Two planes that never touch. Reading the catalog is high-volume, anonymous, cacheable. Writing to it is low-volume, identity-bearing, needs review. Fuse them and you're forced into an always-on server with auth in your read path — the exact cost local-first avoids. So we split them.
  • Git is the database. The canonical catalog is a git data repo. It already gives versioning, diffs, blame, an audit log, and one-command rollback. No Neon/Postgres mirror — that would be a second copy of the source of truth with no read benefit (clients read static files, never a DB).
  • One small server, and only for friction. A stateless Cloudflare Worker exists for a single reason: so contributors don't need a GitHub account and we never ship a write token. It turns an in-app "Submit" into a pull request.
  • GitHub is the moderation tool. Every contribution is a PR. Review is a diff; approval is the merge button; the admin UI, queue, and audit trail are already built and free.

1The two-plane model

The Read plane ships the catalog to every client as static files. The Write plane turns a user's device into a reviewed pull request. They meet in exactly one place: a merge triggers the publish CI, which regenerates the static files the read plane serves.

Read plane · free & static
Git data repo devices/*.json — canonical source of truth
↓  publish CI compiles & uploads
CDN — GitHub Releases → R2 snapshot + deltas + manifest.json
↓  background sync · ETag / 304
App — local SQLite mirror read-mostly; merged with your library
Write plane · low-volume
App — “Submit” / “Correct” / “Report” pseudonymous install ID · no account
↓  HTTPS POST (queued, retries offline)
Cloudflare Worker validate · rate-limit · blocklist
↓  opens PR via GitHub App token
Pull request on data repo review · merge = publish

The loop closes here: merging a PR runs the publish CI → a new snapshot/delta lands on the CDN → every client picks it up on its next background sync. A bad merge is undone with git revert + republish.

2The catalog data repo

A separate repo (patricktr/sigpath-catalog) so catalog churn stays out of the app's history and PRs are cleanly scoped. One device per file — so a PR diff is human-readable and dedup is just a path check. 

sigpath-catalog/
  devices/
    sony/bravia-x90l.json        # one DeviceModel per file; path = stable id
    apple/tv-4k.json
    ...
  connectors/connectors.json     # the controlled vocabulary, versioned with the catalog
  schema/device.schema.json      # JSON Schema the CI validates every submission against
  LICENSE                        # CC0-1.0 — makes the bundled snapshot redistributable

The device shape is already designed for this: DeviceModel in src/schema/device.ts is described in-code as “the unit that gets shared/community-contributed,” and connectors are kept as open string ids precisely so they “serialize cleanly into the future community database.”

3Distribution & efficient updates

The whole catalog is tiny — a DeviceModel is ~300–500 bytes of JSON, so even 5,000 devices is well under 1 MB gzipped. That shapes the update strategy: don't over-build diffing before it pays off, but leave the seam for it.

v1 — whole snapshot behind an ETag

Fetch the gzipped snapshot with a conditional request. A no-op update costs one cheap 304 Not Modified — the CDN's ETag is your free “diff” at this size.

Later — static row-level deltas

Give every row a contentHash and the catalog a monotonic rev. A small manifest lets a client jump from its current rev to head by applying upserts and deletes to its local SQLite. Deltas are static files the publish CI emits — no server computes anything. Cold installs always have a recent full snapshot. 

// manifest.json — the only file the client polls
{
  "rev": 142,
  "minAppVersion": "0.4.0",        // gate catalogs that need newer connectors
  "full":   { "url": "catalog-142.sqlite.gz", "sha256": "…", "bytes": 812345 },
  "deltas": [ { "from": 140, "to": 142, "url": "delta-140-142.json", "bytes": 2310 } ]
}

// delta-140-142.json
{ "from": 140, "to": 142,
  "upserts": [ /* full DeviceModel rows */ ],
  "deletes": [ "sony/old-model-id" ] }
New dependencies. The app has no database or HTTP client today (Cargo.toml carries only dialog + opener). Phase 4 adds tauri-plugin-sql (or bundled rusqlite) for the local mirror and tauri-plugin-http for sync. The community catalog becomes a read-mostly SQLite layer, merged in loadCatalog() alongside the built-ins and the user's personal library.

Where it's hosted

GitHub Releases first (the data repo's CI attaches snapshot + deltas as release assets — zero new infrastructure, genuinely free egress), with Cloudflare R2 as a drop-in upgrade when a custom domain or finer cache control is wanted. The client hits a constant URL either way; the host is an implementation detail it never sees.

4Contributing a device

The “Submit to community” button lives at the end of the existing create wizard (CreateWizard.tsx) and on any custom device in the library. The contributor needs no GitHub account; the app holds no secrets.

  1. App builds a submission. The serialized DeviceModel plus a pseudonymous install ID, app version, and a CC0 license affirmation.
  2. POST to the Worker. Queued locally and retried if offline — submitting never blocks the user or needs connectivity right then.
  3. Worker gatekeeps. JSON-Schema validate → size cap → rate-limit per install ID + IP → blocklist check. Garbage and abuse die here, before touching GitHub.
  4. Worker opens a PR. Using a server-held GitHub App token, it writes devices/<slug>.json on a new branch and labels it new-device.
  5. CI validates the PR. A GitHub Action re-checks schema, connector vocabulary, size, and the dedup key, then comments a rendered preview.
  6. Human merges. Merge fires the publish CI → new snapshot/delta → every client syncs it. Done.
// app → Worker
{
  "kind": "new",
  "installId": "a1b2…",            // random, first-run; never a real identity
  "appVersion": "0.4.0",
  "license": "CC0-1.0",           // contributor affirms original / permissive
  "device": { /* DeviceModel */ }
}

5Correcting & reporting

This falls out of an existing design choice for free: placed devices embed a snapshot of their model (DeviceInstance.model), so a community correction can never break anyone's saved diagram. The source field ("community" | "custom" | "builtin") is the seam.

  1. Edit forks the device. Editing a source:"community" device copies it into the personal library as source:"custom", stamped with forkedFrom: { id, baseRev, baseHash }. The user's diagrams now use their fork.
  2. “Submit correction” diffs against the base. Because we recorded what it was forked from, the app sends a field-level diff (“HDMI 2” → “HDMI 2 (eARC)”, +Optical out), not a blind overwrite. The PR shows reviewers exactly what changed.
  3. Same pipeline as a new device. A correction is just a submission carrying a forkedFrom pointer; the Worker labels it correction.

And a lighter escape hatch: “Report error” (kind:"report" → a GitHub issue) for the common case where a user spots that something's wrong but won't sit down to fix it. It feeds the same queue without requiring an edit.

6Moderation & trust tiers

Because every contribution is a PR, moderation is review-and-merge — but volume is managed with tiers so trivial fixes don't bottleneck on a human and risky changes always do.

SubmissionFrom a new / anonymous IDFrom a trusted contributor
Report (issue)Auto-filed to the queue; no merge risk.
Correction (field diff)Held for review.Eligible for auto-merge once CI is green.
New deviceHeld — full review.Held — full review (new identity still gets a look).

“Trusted” is earned, not logged-in: an install ID with N merged, un-reverted contributions. Everything stays revertible — the catalog's whole history is in git.

7Identity & abuse — without accounts

You asked whether this needs a Neon DB + login to track bad actors. It doesn't — not to start, and not to mirror the catalog. Identity stays as light as the job allows.

LevelWhat it isWhen
Pseudonymous install IDRandom id minted on first run (optionally tied to a hashed email if the user wants credit). Worker rate-limits per ID + IP and keeps a blocklist.v1 — recommended. No login UI at all.
Optional GitHub sign-inOpt-in OAuth, only for contributors who want their handle credited or to earn trusted auto-merge.Additive, later.
Full accountsManaged auth + reputation store.Only if abuse volume ever demands it.

The only state living outside git is a tiny Cloudflare KV/D1 table of moderation state — install IDs, rate counters, a blocklist, submission status. That is a moderation concern, never a copy of the catalog. Text fields are sanitized (the app already depends on dompurify) and payloads are size-capped at the Worker.

8The problems worth naming now

These aren't infrastructure, but they decide whether the community library is any good. The first and the last are the load-bearing ones.

Data license

A redistributable community catalog needs an explicit license so contributions can legally ship inside the bundled snapshot, and so we steer clear of scraped-manufacturer-data IP problems. The whole “win on community, not catalog scale” moat rests on this being clean.

Recommend: CC0-1.0, affirmed at submit time

Canonical identity & dedup

Two people submit “Sony BRAVIA X90L.” Without a normalized key and a merge-don't-duplicate rule, the catalog fills with near-duplicates and rots.

Recommend: slug key mfr/model = the file path

Connector-vocabulary coupling

Connectors are a controlled vocabulary. A device referencing a connector an older client doesn't know must degrade gracefully — which it already does in cableColor() / checkPortCompatibility(). Catalogs that need a new connector are gated by minAppVersion.

Mostly handled — keep the fallback paths honest

Seeding the cold start

The roadmap already says the real hard problem is seeding the catalog, not hosting. An empty contribution system is worthless. A bulk CSV/JSON import path (and a first pass of common gear) is a required workstream.

Highest-risk item — plan it explicitly

Offline & failure behavior

Sync is background and optional; the app works forever on the bundled snapshot if the network or CDN is gone. Submissions queue locally and retry. Local-first all the way down.

Design invariant, not a feature

Versioned & reversible

Because the catalog is git, a poisoned or wrong merge is git revert + republish. The snapshot's rev moves forward; clients converge on the next sync.

A safety property we get for free

9The schema seam (Phase 1 groundwork)

The roadmap promises the schema is “designed for this in Phase 1.” These are the small, additive fields that leave the seam open — documented here, implemented when Phase 4 lands. Nothing here breaks existing .sigpath files. 

// src/schema/device.ts — additive, all optional
type DeviceModel = {
  // …existing fields (id, manufacturer, model, category, ports, source, …)
  rev?: number;             // catalog revision this row came from
  contentHash?: string;     // row hash — powers deltas & dedup
  forkedFrom?: {            // set when a community device is forked to correct it
    id: string;
    baseRev: number;
    baseHash: string;
  };
};

source already exists and is the merge seam; loadCatalog() grows a community-SQLite layer beside the built-in models and the personal library it reads today.
