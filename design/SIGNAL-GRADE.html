sigpath

Signal grade vs cable bandwidth — architecture. Why a matching connector doesn't prove a working link, and how sigpath grows its compatibility primitive from “connectors match” to “connectors match and the rating ≥ the signal's demand.” The design behind roadmap item p1-signalgrade.

Last updated June 22, 2026 · ← Product roadmap · Community architecture → · github.com/patricktr/sigpath

The one idea

  • A connector proves the shape, not the throughput. Both ends are BNC, but a 3G-SDI cable will not pass a 12G-SDI signal. The validator today stops at “the plugs mate.” That's necessary, not sufficient.
  • Capability and demand are different things. A 12G port is capable of 12G; it does not demand it — it will happily emit 3G. The link works whenever demand ≤ capability at every hop. The asymmetry is real: a higher grade always carries a lower one; never the reverse.
  • The show format sets the demand. AV folks describe a show as 1080p59.94 or 2160p59.94, not in Gbps. That one project setting is the demand ceiling — and it's what stops a 12G-capable port from nagging for 12G cable across an HD show.
  • Unknown ⇒ silent. No format ⇒ ask. When a grade is unknown, the check is skipped — turning this on never floods old diagrams with errors. When graded gear exists but no show format is set, the validator doesn't guess: it asks the user to pick one.

1The two-gate model

Compatibility grows a second gate. Gate 1 is today's connector check — does the plug physically mate? Gate 2 is new — is the run's demand within the capability of the cable and the sink? Gate 2 only runs if Gate 1 already passed, and only on connectors that have a bandwidth grade (SDI, HDMI, DisplayPort, USB, Ethernet — never analog audio or power).

Gate 1 · connectors mate · today
Same connector → straight cable checkPortCompatibility() — unchanged
↓  same family / group → adapter or converter
Different group → incompatible BNC↔BNC always passes here
Gate 2 · grade · new
Resolve the run's demand min(source max, show format) — or a per-run override
↓  compare by rank, within one scale
demand ≤ cable rating & demand ≤ sink max under either ⇒ hard error

Why split it: keeping Gate 1 a pure property of two connectors means its three callers — the validator, the cables/adapters list, and the converter finder — keep working untouched. Grade rides alongside as an independent second check, computed where the run and the project are both in scope.

2The 12G→3G→12G gotcha, resolved

The whole design turns on one scenario: a 12G-capable output, a 3G-rated cable, a 12G-capable input — all BNC. Whether that's broken depends entirely on what's flowing, which is the show format. The same diagram is fine for an HD show and broken for a 4K60 show, and sigpath should say so.

Run (all BNC)Show formatDemandCable 3GSinkVerdict
12G out → 3G cable → 12G in 1080p59.94 3G-SDI ≥ ok ≥ ok Pass — silent
12G out → 3G cable → 12G in 2160p59.94 12G-SDI 3G < 12G ≥ ok Error — cable under-rated
12G out → 12G cable → 3G-only in 2160p59.94 12G-SDI ≥ ok 3G < 12G Error — input can't accept
any graded run not set — pending — — — Ask — “pick a show format”

This is the answer to “do we need the user to set a max data rate?” — yes, and frame size/rate (2160p59.94) is the right surface, not raw Gbps. Without it, demand would have to fall back to each source's own maximum, and every 12G port would demand 12G cable everywhere: the false-positive avalanche. With it, an HD show demands only 3G and the gotcha evaporates.

3Where demand comes from

Demand is per-run, but you almost never set it per-run. It cascades from one project setting, with two escape hatches for how real shows actually behave. 

// Resolve the grade a run actually carries (the "required" rating).
function demand(run, project): GradeId | undefined | PENDING {
  const scale = gradeScaleOf(run.source.connector);
  if (!scale) return undefined;            // ungraded family (XLR, IEC…) — Gate 2 skipped

  if (run.signalGrade) return run.signalGrade;  // per-run override (the one 12G feed in an HD show)

  const ceiling = project.signalProfile?.targets?.[scale]
              ?? videoFormatToGrade(project.signalProfile?.videoFormat, scale);
  if (!ceiling) return PENDING;             // graded gear but no show format ⇒ ask the user

  const sourceMax = run.source.grade;          // a source can't emit beyond its own max…
  return sourceMax ? minByRank(sourceMax, ceiling) : ceiling;  // …nor beyond the show
}
  • Per-domain, not one global number. A show is “1080p59.94” for the image domain (SDI + HDMI + DisplayPort move together), but its network is independently 1G/10G and USB is its own thing. The profile is one video format → image grades, plus optional explicit targets for network and USB/data.
  • Per-run override. Mixed-rate shows are normal — the PGM feed is 2160p59.94 (12G) while confidence and multiviewer feeds are 1080i (HD). The project sets the default; connection.signalGrade overrides a single run up or down.
  • Clamped by the source. A 3G camera in a 4K show can only ever emit 3G, so its runs demand 3G — and (phase D) the camera itself earns a soft “below show format” note, not a cable error.

4The grade scales

A grade is meaningful only within a family — you never compare an SDI grade to an HDMI grade. Each family is an ordered ladder; comparison is by rank. The gbps is carried for display and sorting, and for the rare cross-family converter reasoning.

sdiSD‹HD 1.5G‹3G‹6G‹12G‹24G
hdmi1.4 · 10.2G‹2.0 · 18G‹2.1 · 48G
displayport1.2 · 21.6G‹1.4 · 32.4G‹2.0 · 80G
usb2.0 · 480M‹5G‹10G‹20G‹40G‹80G
ethernet100M‹1G‹2.5G‹5G‹10G‹25G

Each connector points at its scale via one new optional field. SDI/BNC → sdi; HDMI / Mini / Micro → hdmi; DisplayPort / Mini-DP → displayport; USB-C/A/B / Thunderbolt → usb; RJ45 / etherCON → ethernet. Everything else (XLR, TRS, RCA, IEC, NEMA, RS-232…) has no scale and is never grade-checked.

5The schema changes

Additive and optional, in the spirit of the community seam. Nothing here breaks an existing .sigpath file — an old file simply has no signalProfile, so Gate 2 stays dormant until a format is set. 

// src/schema/grades.ts — NEW: per-family bandwidth ladders
type GradeScaleId = "sdi" | "hdmi" | "displayport" | "usb" | "ethernet";
type GradeId = string;                         // e.g. "sdi-3g", "hdmi-2.1", "eth-10g"
type GradeTier = { id: GradeId; label: string; rank: number; gbps: number };
// GRADE_SCALES[scale] = ordered GradeTier[]; compare by rank, only within a scale.

// src/schema/connectors.ts — ConnectorDef gains one optional field
gradeScale?: GradeScaleId;            // omitted ⇒ ungraded; never grade-checked

// src/schema/device.ts — Port gains its capability ceiling
grade?: GradeId;                      // max this jack emits/accepts; omitted ⇒ unrated

// src/schema/connection.ts — the run carries the cable's rating + an optional demand override
cableGrade?: GradeId;                 // the cable's SUPPORTED rating
signalGrade?: GradeId;                // per-run demand override (mixed-rate shows)

// src/schema/document.ts — Project gains the demand ceiling
signalProfile?: {
  videoFormat?: string;               // "1080p59.94" | "2160p59.94" → image domain
  targets?: Partial<Record<GradeScaleId, GradeId>>;  // explicit network / usb
};
// SIGPATH_SCHEMA_VERSION: 1 → 2  (forward migration: old files load with profile undefined)

One ripple to plan for: deviceContentHash() must fold port.grade into its canonical — a 12G port and a 3G port are different identities. That recomputes every hash once, so the first catalog publish after this lands looks like “every row changed.” Handle it as a single intentional rev bump, not a sync bug.

6Validation & the “pick a show format” state

Gate 2 adds two issue types, and one project-level state. The two issues ride the existing validation rails — the panel, the red/amber edge styling, the status pill — so no structural UI change is needed for them.

ConditionSeverityMessage
Input can't carry signal grade
sink.grade < demand		 Error “{sink} maxes at 3G-SDI; this run carries 12G-SDI.” The destination physically can't accept it.
Cable under-rated
cableGrade < demand		 Error “Cable rated 3G-SDI; this run carries 12G-SDI — use a 12G-rated cable.” The most fixable failure, but still a failure.
Source below show format phase D
source.grade < target		 Warning “{source} tops out at 3G-SDI; the show is 2160p59.94 (12G).” The source can't make the format — not a cable problem.
Needs show format
hasGradedRuns && !format		 Prompt One banner at the top of the validation panel, plus a distinct status-dot color. Not per-edge — a single call to action.

The needs-format state, precisely

  • Trigger: the diagram has at least one run on a graded scale and no signalProfile.videoFormat (and no explicit target for that scale). A purely analog/power diagram never triggers it — there's nothing to grade.
  • Status dot: today's pill in the status bar is green (ok) / amber (warn) / red (error). This adds a fourth state. Recommendation: a blue “needs setup” dot rather than orange — amber already means “warnings,” and a second orange would be ambiguous. It reads as actionable-but-not-broken.
  • Banner copy (near the top of the validation panel, draft for wording): “Set a show format to check signal grades — sigpath can't tell a 3G cable from a 12G one until it knows what the show runs at.” with a button that opens the format selector.
  • Once set: the banner clears and real per-edge grade errors (if any) appear normally.

7Data entry & the catalog ripple

The roadmap's “every port needs a grade” is the bulk of the work — but the catalog already carries most of the answer as free text. Of 840 ports across 60 models, 612 have a note and ~106 already state a grade ("3G-SDI", "USB 2.0", "HDMI 2.0 Type A", "10/100/1000/10G BASE-T"). The graded families dominate: SDI/BNC 258 ports, HDMI 91, RJ45 72, USB 67.

  1. Parse notes → structured grade. A one-time script over communityCatalog.json lifts the ~106 explicit hints into port.grade. Free coverage on the highest-value ports.
  2. Connector-era defaults for the rest. Where no hint exists, infer a conservative per-connector default (bare SDI/BNC → 3G, the modern baseline; RJ45 → 1G). Corrections flow back through the community “Correct” path.
  3. Grade dropdown in the create wizard. A new column in the port grid that only appears when the chosen connector has a scale, options drawn from that scale. Surfaced read-only in the Inspector.
  4. Project show-format selector. One control (by Validate / Lists, or in project settings). The highest-leverage element in the whole feature — it's what makes Gate 2 usable instead of noisy.
  5. Per-run cable grade + override. In the edge inspector / contextbar — set the cable's rating, or override the demand for this one run.
  6. JSON Schema. device.schema.json (the Phase-4 community CI gate) gains the optional grade enum per scale, so contributions are validated.

Decided: an under-rated cable blocks the run like any other mismatch. It genuinely won't pass the signal reliably; “easiest to fix” doesn't mean “not broken.”

Format → grade table

videoFormatToGrade() needs a curated map (1080p59.94 → SDI 3G + HDMI 2.0; 2160p59.94 → SDI 12G + HDMI 2.1; etc.). Small, but it's the semantic heart — worth getting exactly right.

phase B

Default backfill confidence

Inferred defaults will sometimes be wrong (a “bnc” port that's really HD-only). Acceptable because the community Correct path exists — but worth a “grade: inferred vs stated” flag if corrections lag.

phase C

Grade-aware converters

An up-converter (3G→12G) is a real, purchasable box. Once grades exist, the converter finder can suggest the right-rated one — a natural extension of the existing finder.

phase D
sigpath — local-first AV signal-flow. This document specs roadmap item p1-signalgrade (“Signal grade vs cable bandwidth — connector isn't enough”). Companion to the roadmap and the community architecture.
